# Architecture Decision Record: Challenge To YOU

## ADR-001: Desktop-First with Godot 4

**Status**: Accepted  
**Date**: 2026-07-10

### Context
We need to build a roguelike hacking game with:
- Multi-layer code execution (Frankenstein code)
- WASM sandbox for secure execution
- Multiple eras with different visual themes
- Procedural generation

### Decision
Use **Godot 4** as the frontend engine, targeting **desktop platforms** (Windows, Mac, Linux).

### Rationale
1. **WASM Performance**: Browser-based WASM sandbox would be too slow for multiple simultaneous executions. Desktop gives raw horsepower.
2. **GDExtension Support**: Go code can be compiled as shared library and linked directly into Godot, eliminating network latency.
3. **Cross-Platform**: Godot handles Windows/Mac/Linux builds natively.
4. **Offline Capability**: Desktop app works without internet.

### Consequences
- No web version for MVP (can add later)
- Must distribute via Steam/Itch.io (not browser)
- Higher initial setup complexity

## ADR-002: Go Backend with WASM Sandbox

**Status**: Accepted  
**Date**: 2026-07-10

### Context
Player code must execute securely without:
- Crashing the game
- Accessing system resources
- Infinite loops blocking execution

### Decision
Use **Go** for backend logic with **Extism/Wasmer** for WASM sandbox execution.

### Rationale
1. **Concurrency**: Go's goroutines handle multiple simultaneous code executions effortlessly.
2. **Security**: WASM provides complete isolation - player code cannot escape sandbox.
3. **Performance**: WASM executes in microseconds (vs Docker's seconds).
4. **Flexibility**: Can support multiple languages (custom DSL, Python, JS) via different WASM modules.

### Consequences
- Must learn WASM integration (Extism)
- Initial setup more complex than simple text parsing
- Long-term scalability is excellent

## ADR-003: Procedural Generation with Luck Mechanic

**Status**: Accepted  
**Date**: 2026-07-10

### Context
Fixed puzzles become repetitive. Need infinite replayability.

### Decision
Use **seed-based procedural generation** with a **Luck stat** that affects difficulty.

### Rationale
1. **Infinite Content**: 15-20 modular code segments can produce thousands of unique challenges.
2. **Replayability**: Luck mechanic creates variety - some runs are easier, some harder.
3. **Roguelike Appeal**: Players chase "good luck" runs, creating streaming content.
4. **Scalability**: Adding new code segments instantly expands the pool.

### Consequences
- Must ensure at least one solution always exists
- Luck distribution must be fair (not too punishing)
- Need to test for invalid combinations

## ADR-004: Multi-Era Progression (2 Eras in v1)

**Status**: Accepted  
**Date**: 2026-07-10

### Context
Single era becomes stale. Need to prove "multiverse" hook.

### Decision
Launch with **2 eras**: Medieval Magitech (Tier 1) → Cyberpunk Neon (Tier 2).

### Rationale
1. **Proof of Concept**: Two eras prove the multiverse mechanic works.
2. **Visual Contrast**: Magitech (dark/mystical) vs Cyberpunk (neon/terminal) shows range.
3. **Code Progression**: Custom DSL (runes) → Real scripting (Python/JS) feels like advancement.
4. **Manageable Scope**: 2 eras is achievable in 1 month.

### Consequences
- Must build era-specific UI themes
- Need DSL parser for Magitech era
- Cyberpunk era uses real languages (easier)

## ADR-005: Three Core Gameplay Modes

**Status**: Accepted  
**Date**: 2026-07-10

### Context
Single gameplay mode becomes repetitive. Need variety.

### Decision
Launch with **3 modes**: Architect, Ghost, Saboteur.

### Rationale
1. **Full Spectrum**: Building (Architect) → Stealth (Ghost) → Breaking (Saboteur) covers all hacker fantasies.
2. **Skill Variety**: Each mode tests different skills (planning, optimization, judgment).
3. **Replay Value**: Players can focus on preferred mode or master all three.
4. **Content Efficiency**: Same code segments work for all modes (different objectives).

### Consequences
- Must implement tracking meters for Ghost mode
- Need chain reaction detection for Saboteur mode
- Architect mode is baseline (easiest to implement)

## ADR-006: Local AI (Ollama) vs Cloud API

**Status**: Accepted  
**Date**: 2026-07-10

### Context
AI must analyze code style and generate passcodes without:
- High API costs
- Network latency
- Privacy concerns

### Decision
Use **local LLM (Ollama + Llama 3)** with **AST parsing** fallback.

### Rationale
1. **Cost**: Local execution is free (vs $0.01-$0.10 per API call).
2. **Privacy**: Player code never leaves their machine.
3. **Speed**: Local inference is faster than network round-trip.
4. **Offline**: Game works without internet.

### Consequences
- Requires Ollama installation (or bundled)
- Local model less powerful than GPT-4
- AST parsing provides deterministic fallback

## ADR-007: Skip WASM for Itch.io Alpha

**Status**: Accepted  
**Date**: 2026-07-10

### Context
1-month deadline is tight. WASM integration adds complexity.

### Decision
**Skip WASM sandbox** for Itch.io alpha. Use **Go text parsing** instead.

### Rationale
1. **Speed**: Text parsing is simpler and faster to implement.
2. **Validation**: Alpha tests game mechanics, not sandbox security.
3. **Iterate**: Add WASM for Steam Early Access based on player feedback.
4. **Survival**: Must ship in 30 days or die.

### Consequences
- Alpha is less secure (player code can crash game)
- Must add timeout/limits manually
- WASM becomes priority for Phase 2

## ADR-008: Itch.io First, Steam Later

**Status**: Accepted  
**Date**: 2026-07-10

### Context
Need to validate game before charging money.

### Decision
Launch **free/PWYW alpha on Itch.io** (Week 4), then **Steam Early Access** (Month 3).

### Rationale
1. **Feedback**: Free alpha attracts more testers.
2. **Wishlists**: Itch.io traffic drives Steam wishlists.
3. **Protection**: Early Access protects from bad reviews.
4. **Revenue**: Steam generates income for full development.

### Consequences
- Must set up Steam page in Week 1 (30-day approval)
- Itch.io is free, no revenue initially
- Steam takes 30% cut

## ADR-010: Document Everything

**Status**: Accepted  
**Date**: 2026-07-10

### Context
Solo dev with fluid team needs clear documentation for onboarding.

### Decision
**Document all conversations, decisions, and changes** in structured format.

### Rationale
1. **Onboarding**: New teammates can quickly understand context.
2. **Accountability**: Decisions are tracked with rationale.
3. **History**: Can reference why decisions were made.
4. **AI Collaboration**: Provides context for AI assistants.

### Consequences
- Must maintain `docs/CONVERSATIONS/`, `docs/DECISIONS/`, `docs/CHANGES/`
- Adds overhead to development
- Essential for team scaling
